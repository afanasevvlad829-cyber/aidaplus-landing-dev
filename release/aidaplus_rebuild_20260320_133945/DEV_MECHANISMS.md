# DEV Mechanisms

Этот файл фиксирует временные `dev`-механики и нестандартные обходные решения, которые были добавлены в проект во время доработки лендинга.

Цель:
- не забыть, что именно сейчас включено;
- быстро пройтись по списку перед финальной боевой выкладкой;
- понимать, какие функции являются временными, а не частью целевой production-архитектуры.

## 1. Единый рубильник попапов

Файл:
- `index.html`

Что сделано:
- добавлен глобальный флаг:
  - `window.AC_POPUPS_ENABLED = false;`
- если флаг выключен, документ получает класс:
  - `ac-popups-disabled`
- через него скрываются все сценарные попапы и плашки.

Сейчас скрываются:
- `#acReturnBanner`
- `#acFixBackdrop`
- `#acBar`
- `#acNudgeProject`
- `#acNudgeGuarantee`
- `#acExitBackdrop`
- `#acRevisitBackdrop`
- `#acSearchBackdrop`

Зачем:
- чтобы тестировать верстку без помех от сценарных всплывашек.

Перед финальной боевой выкладкой:
- решить, должны ли попапы вернуться;
- если да, переключить:
  - `window.AC_POPUPS_ENABLED = true;`

## 2. Закомментированные сценарные попапы в legacy-скриптах

Файл:
- `index.html`

Что временно отключено комментариями:
- автопоказ nudges после выбора возраста;
- запуск `deterministicPopups();`

Зачем:
- даже при наличии legacy-кода не запускать сценарные маркетинговые плашки во время визуального тестирования.

Перед финальной боевой выкладкой:
- проверить, нужен ли возврат именно этой логики;
- если нужен, раскомментировать вручную.

## 3. Отключенная диагностика карточек смен

Файл:
- `src/scripts/main.js`

Что сделано:
- вызов `startShiftsDiagnostics();` закомментирован.

Зачем:
- диагностика использовалась для отлова момента, когда карточки смен сужались и перезаписывались legacy-правилами.

Перед финальной боевой выкладкой:
- оставить выключенной;
- при необходимости временно включать только для отладки.

## 4. Dev-обновление команды с сайта

Файл:
- `src/scripts/main.js`

Флаг:
- `AC_DEV_REMOTE_TEAM_REFRESH_ENABLED = true`

Что делает:
- при каждой загрузке страницы команда подгружается заново с `https://aidacamp.ru/media`
- используется:
  - `fetch(..., { cache: "no-store" })`
- есть повторные попытки после старта;
- есть периодическое обновление раз в минуту.

Зачем:
- блок `Команда` не всегда подхватывал свежие фото и описания;
- это временный способ форсировать актуальные данные при разработке.

Риски:
- зависимость от внешней HTML-структуры;
- лишний сетевой запрос;
- неидеально для production-статики.

Перед финальной боевой выкладкой:
- лучше заменить на стабильный источник:
  - `team.json`
  - локальный манифест
  - или выключить dev-refresh
- для отключения выставить:
  - `AC_DEV_REMOTE_TEAM_REFRESH_ENABLED = false`

## 5. Dev-подстановка видео из Rutube

Файл:
- `src/scripts/main.js`

Флаг:
- `AC_DEV_RUTUBE_VIDEO_FEED_ENABLED = true`

Что сделано:
- добавлен массив:
  - `AC_DEV_RUTUBE_VIDEOS`
- видео рендерятся во вкладке `Видео` через `iframe` embed Rutube;
- при загрузке страницы этот список подставляется автоматически.

Текущие видео:
- `https://rutube.ru/shorts/f1538387b19f82f0305f7ae7222bf57d/`
- `https://rutube.ru/shorts/01ca8a077f86db0b95bb0adfebd8ebce/`
- `https://rutube.ru/shorts/41bbae1d1b167cd49c7aad94cc76b133/`

Зачем:
- ограничение Tilda по размеру вертикальных видео;
- быстрый способ показать реальные ролики без локального хранения `.mp4`.

Риски:
- зависимость от внешней платформы;
- embed тяжелее и менее контролируем по стилю, чем локальное видео.

Перед финальной боевой выкладкой:
- решить окончательно:
  - оставляем Rutube,
  - переносим на сервер / storage,
  - или формируем отдельный video-manifest
- если не нужен dev-режим, выставить:
  - `AC_DEV_RUTUBE_VIDEO_FEED_ENABLED = false`

## 6. Жесткие layout-фиксы карточек смен

Файлы:
- `src/styles/main.css`
- `src/scripts/main.js`

Что делалось:
- жёсткая стабилизация ширин;
- повторное принудительное применение layout после `renderShifts`;
- отдельная правая колонка для цены;
- перестройка выбранной карточки в более стабильную сетку.

Зачем:
- legacy-скрипты и каскад стилей перезаписывали размеры карточек;
- цена и элементы карточки наезжали друг на друга.

Важно:
- это не один флаг, а набор стабилизационных слоёв;
- перед production имеет смысл ещё раз пройтись и убрать лишние костыли, если финальная верстка станет стабильной.

## 7. Техническая версия в интерфейсе

Файл:
- `index.html`

Что сделано:
- есть meta:
  - `ac-tech-version`
- есть видимый badge:
  - `TECH v...`

Зачем:
- быстро понимать, какой HTML реально открыт в браузере;
- диагностировать кэш.

Это полезно и для production, но:
- можно оставить;
- можно скрыть, если не хотите показывать пользователю.

## Рекомендуемый чек перед production

Пройтись по этим пунктам:

1. Попапы:
- решить, включаем или нет:
  - `AC_POPUPS_ENABLED`
- проверить, нужно ли возвращать legacy `deterministicPopups()`

2. Команда:
- решить, оставляем ли `remote refresh`
- лучше заменить на стабильный источник данных

3. Видео:
- решить, остаёмся на Rutube или переносим в постоянное хранилище

4. Диагностика:
- убедиться, что `startShiftsDiagnostics()` остаётся выключенной

5. Карточки смен:
- проверить, какие из жёстких фиксов действительно ещё нужны

6. Технический badge:
- оставить или убрать из пользовательского интерфейса
